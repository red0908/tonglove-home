## 背景与目标

### 这个升级要解决什么问题

官网 `/ai` 页面已有两个 Demo 上线，但都存在明显短板：


| Demo                 | 当前问题                                              |
| -------------------- | ------------------------------------------------- |
| Demo 1：智能 RAG 问答     | 只做关键词词频匹配，没有语义理解，"你有哪些项目经验"无法匹配"项目案例"文档           |
| Demo 2：虚拟 Agent 话术生成 | 前端传 `decision_maker`，后端只识别中文`决策者`，角色枚举 Bug 导致功能失效 |


全局悬浮 AI 助手（`AiChatWidget.tsx`）复用同一套有缺陷的 RAG 接口，问答质量弱。

### 升级后的目标

> **将官网打造成一个"会介绍自己"的 AI 站点**：访客可以用自然语言问任何关于我的问题，AI 基于真实内容精准回答，同时展示 RAG + Dify 工程能力。

具体功能目标：

- 用 Dify 知识库 + 向量检索替换现有关键词 RAG
- 知识库覆盖官网所有核心内容（项目、文章、关于我、专利等）
- 修复 Agent Demo 角色枚举问题
- 支持多轮对话上下文（当前为单轮）
- 悬浮助手升级为更智能的问答体验
- 管理后台支持一键同步官网内容到知识库（可选，Phase 4）

## 现有代码缺陷分析

### RAG 检索核心问题

```python
# 现有检索：简单关键词词频匹配（backend/routers/ai.py）
def _keyword_search(question: str, docs: list, top_k: int = 2):
    q_words = set(question.lower().split())
    score = sum(1 for w in q_words if w in text)
```


| 问题                | 影响                     |
| ----------------- | ---------------------- |
| 无语义理解，只做词频匹配      | "你有哪些项目经验"无法匹配"项目案例"文档 |
| Top-K 固定为 2，无阈值过滤 | 不相关文档也会被拼入 context     |
| 无 Embedding 向量检索  | 近义词、同义句无法命中            |
| Agent Demo 角色 Bug | 前端传英文枚举，后端只识别中文        |
| 知识库内容依赖手动录入       | 官网信息更新后知识库不同步          |

## 技术架构设计

### 整体数据流

```
用户提问
    │
    ▼
前端（React）
    │  POST /api/v1/ai/rag
    ▼
FastAPI 后端（ai.py）
    │  HTTP 调用 Dify API
    ▼
Dify 平台
    ├─ 知识库检索（向量检索，Embedding 模型）
    │      └─ 召回 Top-K 相关文档片段
    ├─ LLM 生成（DeepSeek API）
    │      └─ 基于检索结果生成回答
    └─ 返回 answer + sources
    │
    ▼
FastAPI 格式化响应
    │
    ▼
前端展示答案 + 来源引用
```

### 为什么选 Dify 而不是自建向量库


| 维度            | 自建向量 RAG（Chroma/Weaviate）      | Dify 托管 RAG       |
| ------------- | ------------------------------ | ----------------- |
| 开发成本          | 高（需自写 chunking、embedding、检索管线） | 低（UI 配置即可）        |
| 检索质量          | 取决于自己的实现                       | 开箱即用的混合检索（向量+关键词） |
| 知识库管理         | 需开发后台界面                        | Dify 自带管理 UI      |
| 工作流编排         | 需自建                            | 可视化拖拽             |
| 与 DeepSeek 集成 | 需手动对接                          | Dify 已集成，直接配置     |
| 运维            | 需维护向量数据库                       | Dify Cloud 免运维    |
| **结论**        | 适合深度定制                         | **适合快速落地，本项目首选**  |


> 注意：官网用 Dify 快速落地，和另一个项目 OpenClaw 手写 Chroma 是刻意区分的——形成"会用工具 + 会写底层"的双重能力证明。

### Dify 接入方式

使用**对话型应用 API**（Chatbot 类型）：

```
应用类型：对话型应用（Chatbot）
调用端点：POST https://api.dify.ai/v1/chat-messages
认证方式：Authorization: Bearer {DIFY_API_KEY}
```

## Phase 1：Dify 知识库搭建（Day 1-2）

### 步骤 1：Dify 平台配置

1. 登录 [cloud.dify.ai](https://cloud.dify.ai)，创建新工作区
2. 进入「模型供应商」，配置 **DeepSeek API Key**
3. 配置 Embedding 模型：**优先使用 DeepSeek 兼容的 Embedding**，避免引入额外 API 供应商，Dify 支持直接配置 DeepSeek 作为 Embedding 模型

> 提示：不推荐 `text-embedding-3-small`（OpenAI），会增加一个 API 供应商。用 DeepSeek 统一管理更简单。

### 步骤 2：准备知识库文档

完成 Phase 1 前，需准备以下 Markdown 文档：

```

├── 01_个人简介.md          # 姓名、背景、专业方向、工作经历
├── 02_技术专长.md          # 技术栈、熟悉的框架工具
├── 03_项目案例/

│   └── 项目B.md
├── 04_AI能力介绍.md        # RAG、Dify、DeepSeek、多智能体经验
├── 05_技术专利.md          # 专利名称、摘要、应用场景
├── 06_博客精选.md          # 代表性文章摘要
└── 07_联系与合作.md        # 邮箱、社交媒体、合作意向
```

### 知识库内容分类规划


| 知识库名称     | 内容来源             | 更新频率    |
| --------- | ---------------- | ------- |
| `个人简介`    | About 页面、自我介绍文本  | 低（半年一次） |
| `项目案例`    | Projects 页面所有项目  | 中（新项目时） |
| `博客文章`    | Posts 页面已发布文章    | 高（持续更新） |
| `技术专利`    | 专利描述             | 低       |
| `AI 能力介绍` | AI 体验专区说明、工具使用经验 | 中       |
| `联系与合作`   | 联系方式、合作意向说明      | 低       |


### 步骤 3：文档分块配置（在 Dify 知识库设置中）

```
- 分块方式：按段落 + 最大 Token 限制
- 最大 Token 数：512 tokens / chunk
- 重叠 Token：50 tokens（保持上下文连贯）
- 检索方式：混合检索（向量 + 关键词）
- Top-K：3
- 相关度阈值：0.5
```

4. 发布并获取 API Key

## Phase 2：后端接入 Dify（Day 3）

### 2.1 新增环境变量

在 `backend/.env` 中添加：

```env
DIFY_API_KEY=app-xxxxxxxxxxxxxxxx
```

### 2.2 Schema 扩展（`backend/schemas.py`）

```python
class RAGRequest(BaseModel):
    question: str
    conversation_id: str = ""   # 支持多轮对话

class RAGResponse(BaseModel):
    answer: str
    sources: list[str]
    conversation_id: str = ""   # 回传给前端保持上下文
```

### 2.3 替换 `backend/routers/ai.py` 核心逻辑

将原有的 DeepSeek 直接调用替换为 Dify API 调用：

```python
import httpx
import os

DIFY_API_KEY = os.getenv("DIFY_API_KEY", "")
DIFY_API_URL = "https://api.dify.ai/v1/chat-messages"

@router.post("/rag", response_model=schemas.RAGResponse)
async def rag_query(payload: schemas.RAGRequest, db: Session = Depends(get_db)):
    if not DIFY_API_KEY:
        return schemas.RAGResponse(answer="AI 服务未配置，请联系管理员。", sources=[])
    
    async with httpx.AsyncClient(timeout=60) as client:
        resp = await client.post(
            DIFY_API_URL,
            headers={
                "Authorization": f"Bearer {DIFY_API_KEY}",
                "Content-Type": "application/json"
            },
            json={
                "inputs": {},
                "query": payload.question,
                "response_mode": "blocking",
                "conversation_id": payload.conversation_id or "",
                "user": "website-visitor",
            },
        )
        resp.raise_for_status()
        data = resp.json()
    
    # 提取引用来源
    sources = []
    retriever_resources = data.get("metadata", {}).get("retriever_resources", [])
    for r in retriever_resources:
        if r.get("document_name"):
            sources.append(r["document_name"])
    
    return schemas.RAGResponse(
        answer=data["answer"],
        sources=list(set(sources)),
        conversation_id=data.get("conversation_id", ""),
    )
```

## Phase 3：前端升级（Day 4）

### 3.1 修复 Agent Demo 角色枚举 Bug

**问题根因**：前端传 `decision_maker`（英文），后端只识别 `决策者`（中文）。

**修复方式**：统一改为中文值：

```typescript
// frontend/src/pages/AIPlaygroundPage.tsx
const ROLE_OPTIONS = [
  { value: '决策者', label: '决策者', desc: '关注 ROI、风险与战略价值' },
  { value: '使用者', label: '使用者', desc: '关注易用性、效率与日常体验' },
  { value: '观望者', label: '观望者', desc: '关注技术成熟度与行业趋势' },
]
```

### 3.2 RAG 对话支持多轮

在 RAG Demo 组件中维护 `conversationId` 状态：

```typescript
const [conversationId, setConversationId] = useState<string>('')

const handleAsk = async (question: string) => {
  const resp = await aiApi.rag({ 
    question, 
    conversation_id: conversationId 
  })
  // 保存新的 conversation_id，下次发送时携带
  setConversationId(resp.conversation_id)
  // ... 展示 resp.answer 和 resp.sources
}
```

### 3.3 新增「清空对话」按钮

多轮对话模式下，需要支持重置上下文开启新会话：

```typescript
const handleClearChat = () => {
  setConversationId('')
  setMessages([])
}

// 在 UI 中渲染
<button onClick={handleClearChat} className="text-sm text-gray-500 hover:text-red-500">
  清空对话
</button>
```

### 3.4 优化引用来源展示

Dify 返回的 `retriever_resources` 中含有 `content` 字段（具体片段内容），可升级显示：

```typescript
// 当前：只显示文档标题
{sources.map(src => <span key={src}>{src}</span>)}

// 升级后：显示文档名 + 片段摘要
{sources.map(src => (
  <div key={src.document_name} className="text-xs text-gray-400 mt-1">
    📄 {src.document_name}
    {src.content && <p className="truncate">{src.content.slice(0, 60)}…</p>}
  </div>
))}
```

在管理后台文章/项目编辑器保存时，自动同步到 Dify 知识库：

```
管理后台 → 文章/项目编辑器 → 保存 → 触发同步
     │
     ▼
backend/routers/admin/knowledge_sync.py
     │  调用 Dify Dataset API
     ▼
Dify 知识库（自动更新文档）
```

Dify Dataset API：

```
POST https://api.dify.ai/v1/datasets/{dataset_id}/document/create_by_text
Authorization: Bearer {DIFY_DATASET_API_KEY}
```
