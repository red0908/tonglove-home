# TongAI 官网 · 开发与发布工作流

> 本文档描述如何在本地开发新功能、测试验证，并安全地发布到线上环境。  
> **核心原则：本地开发 → 功能验证 → 推送 master → 自动部署上线**

---

## 一、环境全景

| 维度 | 本地开发环境 | 线上生产环境 |
|------|------------|------------|
| 前端地址 | `http://localhost:5173` | Vercel 分配的域名 |
| 后端地址 | `http://localhost:8001` | Railway 分配的域名 |
| 数据库 | Railway 云端 PostgreSQL（直连） | Railway 云端 PostgreSQL（同一实例） |
| 图片存储 | Cloudinary（同一账号） | Cloudinary（同一账号） |
| 触发部署 | 手动启动 | `git push origin master` 后自动触发 |

> **注意**：本地与生产共用同一个 Railway 数据库实例。本地开发时，`backend/.env` 中的 `DATABASE_URL` 已指向 Railway 云端数据库，操作数据会直接影响线上数据。  
> 如需隔离，参考「附录 A：配置本地独立数据库」。

## 二、日常开发流程（标准迭代）

### 第一步：拉取最新代码

每次开始开发前，先同步远端最新代码，避免冲突：

```bash
cd /Users/zhangruntong/Documents/我的官网2026/tongai
git pull origin master
```

---

### 第二步：启动本地开发环境

需要开两个终端窗口分别启动前端和后端。

**终端 1 — 启动后端：**

```bash
cd backend
source .venv/bin/activate        # 激活 Python 虚拟环境
uvicorn main:app --reload --port 8001
```

**终端 2 — 启动前端：**

```bash
cd frontend
npm run dev
```

启动成功后访问 `http://localhost:5173` 即可看到本地版本。  
前端的 `/api` 和 `/static` 请求会通过 Vite 代理自动转发到 `localhost:8001`，无需手动配置。

---

### 第三步：开发与调试

正常编写代码。前端热更新（HMR）和后端 `--reload` 均已开启，保存文件即时生效。

**常用调试地址：**
- 前端页面：`http://localhost:5173`
- 后端 API 文档（Swagger）：`http://localhost:8001/docs`
- 后端 API 文档（Redoc）：`http://localhost:8001/redoc`

---

### 第四步：提交代码并发布上线

本地验证无误后，推送到 `master` 分支触发自动部署：

```bash
# 查看改动文件
git status

# 暂存所有改动（或精确指定文件）
git add .

# 提交，写清楚本次改动内容
git commit -m "feat: 简明描述你做了什么"

# 推送，触发自动部署
git push origin master
```

**推送后发生的事：**
1. **Vercel** 检测到 `master` 更新，自动拉取代码、执行 `npm run build`，将新前端部署到 CDN，约 1～2 分钟完成。
2. **Railway** 检测到 `master` 更新，自动重新构建后端镜像，执行 `alembic upgrade head` 迁移数据库，然后启动新服务，约 2～3 分钟完成。

---

### 第五步：验证线上效果

部署完成后，访问线上域名确认功能正常。如遇问题，参考「六、线上问题排查」。

## 三、涉及数据库变更时的操作

每次修改了数据库模型（`backend/models.py`），必须生成并提交 Alembic 迁移文件。

### 生成迁移文件

```bash
cd backend
source .venv/bin/activate

# 自动检测模型变化，生成迁移脚本
alembic revision --autogenerate -m "描述本次数据库变更"
```

命令成功后，`migrations/versions/` 下会生成一个新的 `.py` 文件。

### 本地预览迁移（可选）

```bash
# 查看将要执行的 SQL（dry run，不真正执行）
alembic upgrade head --sql
```

### 将迁移文件一并提交

```bash
git add migrations/versions/
git add backend/models.py    # 以及其他改动文件
git commit -m "feat: 添加 xxx 字段，更新数据库结构"
git push origin master
```

Railway 在部署时会自动执行 `alembic upgrade head`，迁移会在新服务启动前完成。

## 四、提交信息规范

良好的提交信息便于日后回溯。推荐使用以下前缀：

| 前缀 | 场景 |
|------|------|
| `feat:` | 新功能 |
| `fix:` | Bug 修复 |
| `style:` | 样式、UI 调整（不影响功能） |
| `refactor:` | 代码重构（不新增功能，不修复 bug） |
| `content:` | 更新网站内容、文案、图片 |
| `chore:` | 依赖更新、配置修改等杂项 |
| `docs:` | 文档更新 |

示例：
```
feat: 新增作品集筛选功能
fix: 修复移动端导航栏遮挡问题
content: 更新关于页个人简介
```

## 五、紧急修复线上 Bug（Hotfix）

当线上出现需要立即修复的 bug 时，正常流程开发太慢，可走以下快速通道：

```bash
# 1. 确保本地是最新代码
git pull origin master

# 2. 在本地快速定位并修复 bug
# ... 修改代码 ...

# 3. 本地验证修复有效后立即推送
git add .
git commit -m "fix: 紧急修复 xxx 问题"
git push origin master

# 4. 等待 Vercel/Railway 自动部署（约 2～3 分钟）
# 5. 线上验证修复结果
```

## 六、线上问题排查

### 查看后端日志（Railway）

1. 打开 [railway.app](https://railway.app) → 进入你的项目
2. 点击后端服务 → 选择 **Deployments** 标签
3. 点击最新部署 → 查看实时日志

常见错误：
- `alembic upgrade head` 失败：说明迁移文件有冲突，检查 `migrations/versions/` 下文件
- `500 Internal Server Error`：后端代码报错，看日志中的 Traceback

### 查看前端部署日志（Vercel）

1. 打开 [vercel.com](https://vercel.com) → 进入你的项目
2. 点击最新部署 → 查看 **Build Logs**

常见错误：
- TypeScript 类型错误导致 `npm run build` 失败
- 环境变量未配置（Vercel 中检查 Settings → Environment Variables）

### 回滚线上版本

**前端回滚（Vercel）：**
1. Vercel 项目页 → Deployments → 找到上一个稳定版本
2. 点击三点菜单 → **Redeploy** → 立即生效

**后端回滚（Railway）：**
1. Railway 项目页 → 后端服务 → Deployments
2. 找到上一个稳定部署 → **Redeploy**

> Railway 回滚**不会**自动回滚数据库迁移。若迁移有破坏性变更，需手动执行 `alembic downgrade -1`（连接生产数据库执行）。

## 七、环境变量管理

### 本地环境变量

文件位置：`backend/.env`（已在 `.gitignore` 中，**不会提交到 git**）

```env
DATABASE_URL=postgresql://...        # Railway 数据库连接串
SECRET_KEY=your-jwt-secret-key
ACCESS_TOKEN_EXPIRE_MINUTES=60
DEEPSEEK_API_KEY=                    # 可选，AI 功能
ALLOWED_ORIGINS=http://localhost:5173
```

### 生产环境变量

在 **Railway 控制台** 管理：
- Railway → 后端服务 → Variables 标签

在 **Vercel 控制台** 管理（如前端有需要）：
- Vercel → 项目 → Settings → Environment Variables

**原则：敏感信息（密钥、数据库密码）只存在于平台控制台，绝不提交到 git。**

## 八、依赖更新

### 前端依赖

```bash
cd frontend
npm update              # 更新到 package.json 中指定的范围内最新版
# 或
npm install <包名>@latest   # 更新指定包到最新版
```

更新后需重新测试，确认无兼容性问题再提交。

### 后端依赖

```bash
cd backend
source .venv/bin/activate
pip install --upgrade <包名>
pip freeze > requirements.txt    # 更新锁定文件
```

同样需本地测试通过再提交。

## 附录 A：配置本地独立数据库（可选）

如果不想本地开发时误操作线上数据，可以用 Docker 启动一个本地 PostgreSQL：

```bash
# 启动本地 PostgreSQL 容器
docker run -d \
  --name tongai-local-db \
  -e POSTGRES_USER=localuser \
  -e POSTGRES_PASSWORD=localpass \
  -e POSTGRES_DB=tongai_dev \
  -p 5432:5432 \
  postgres:15

# 初始化数据库结构
cd backend
source .venv/bin/activate
alembic upgrade head
```

然后将 `backend/.env` 中的 `DATABASE_URL` 改为：

```env
DATABASE_URL=postgresql://localuser:localpass@localhost:5432/tongai_dev
```

开发完成后，切换回 Railway 的连接串再测试生产数据。
